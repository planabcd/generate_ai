# 投标文件生成设计方案（规则版，无LLM）

## 1. 目标与边界

1. 输入：`招标文件（模板） + 手填资料 + 附件`。
2. 输出：`投标文件.docx`。
3. 唯一格式标准：招标文件中“响应文件格式”章节（通常为第五章）。
4. 不使用LLM，不做AI自由生成，不消耗模型额度。
5. `AI生成的文件.docx`仅作为字段参考样例，不作为格式标准。

## 2. 核心原则

1. 结构优先：先解析招标文件中的“响应文件格式”结构，再生成投标文件。
2. 固定文本优先：承诺类、声明类文本按招标文件原文模板替换变量。
3. 事实字段仅来自手填资料/附件，不推测、不编造。
4. 缺失即报：字段缺失、附件缺失、章节缺失必须在报告中明确。

## 3. 输入规范（目录驱动）

`project_dir/`

1. `招标文件/招标文件.docx`（必需）
2. `手填资料/基础信息.json`（必需）
3. `附件/`（按招标要求可选或必选）
4. `参考/AI生成的文件.docx`（可选，仅字段参考）

`基础信息.json`最小字段：
1. `supplier_name`
2. `project_name`（如已知）
3. `project_no`（如已知）
4. `package_no`（如有）
5. `legal_name`
6. `legal_id_no`
7. `doc_date`
8. `signatory_name`

## 4. 处理流程

1. 解析招标文件
- 定位“响应文件格式”章节。
- 提取子章节标题（如：法定代表人授权书、承诺函、技术应答表等）。
- 提取模板正文与表格结构。

2. 建立章节计划（Section Plan）
- 为每个子章节标记类型：
  - `FIXED_WITH_VARIABLES`：固定文本+变量替换
  - `TABLE_FILL`：表格字段填充
  - `ATTACHMENT_SLOT`：附件占位
  - `MANUAL_REQUIRED`：必须人工补充

3. 字段映射
- 手填资料字段映射到章节变量。
- 从附件文件名和已知键值做规则映射（不做OCR、不做LLM）。

4. 文档生成
- 按章节计划顺序输出投标文件。
- 固定章节使用招标文件原文模板，仅替换变量。
- 表格章节按列填充。

5. 质检
- 必填字段是否缺失。
- 必要附件是否缺失。
- 章节是否完整覆盖“响应文件格式”要求。
- 输出 `qa_report.json`。

## 5. 数据模型

`section_plan.json`
1. `section_id`
2. `section_title`
3. `section_type`
4. `source_locator`（来自招标文件的定位）
5. `required_fields`
6. `required_attachments`

`qa_report.json`
1. `missing_required_fields`
2. `missing_required_attachments`
3. `missing_required_sections`
4. `table_unfilled_cells`
5. `notes`

## 6. 模块设计（不改实现，仅设计）

1. `tender_parser`
- 输入：招标文件docx
- 输出：章节树、表格定义、固定模板文本

2. `section_planner`
- 输入：章节树
- 输出：`section_plan.json`

3. `field_mapper`
- 输入：基础信息.json + 附件索引
- 输出：标准字段字典

4. `bid_builder`
- 输入：section_plan + 标准字段 + 固定模板文本
- 输出：`投标文件.docx`

5. `qa_checker`
- 输入：section_plan + 标准字段 + 生成结果
- 输出：`qa_report.json`

## 7. 验收标准

1. 输出文档章节顺序与招标文件“响应文件格式”一致。
2. 固定声明章节文本不漂移，仅变量变化。
3. 所有必填字段缺失项都在报告中可见。
4. 所有必需附件缺失项都在报告中可见。
5. 不调用任何LLM接口即可完成生成。

## 8. 本期不做

1. LLM抽取、LLM文案生成。
2. OCR识别。
3. 对话渠道接入（飞书/微信/Agent）。

## 9. 后续扩展

1. 在规则版通过后，再引入LLM作为可选增强层。
2. 再做对话入口接入，不影响核心生成引擎。
